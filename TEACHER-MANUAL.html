    Platform Setup — Complete Guide

    ⚠
    Do this before June 8. This section is written for you, but also clear enough for a tech-savvy TA or colleague to execute if needed. Follow steps in order — some steps are dependencies for others.
    ℹ
    What students need: A browser. That's it. No installs. No downloads. GitHub Codespaces runs VS Code entirely in the browser. This is the whole reason we chose it — locked-down school MacBook Airs are not a problem.

    A · GitHub Organization setup

    You already have a GitHub Enterprise org for Kinkaid/EMSI. These are the settings to verify before Day 1.

    1. Sign in at github.com with your educator account. Navigate to your org: github.com/[your-org-name]
    2. Verify Codespaces is enabled for the org. Go to Org Settings → Codespaces. Make sure "Enable for all members" is ON. If you don't see the option, it may need to be enabled at the Enterprise level — contact your GitHub admin.
    3. Set Codespace machine type limit. In Org Settings → Codespaces → Machine types, restrict to 2-core only. This prevents students from accidentally spinning up expensive machines. Each student gets 60 free hours/month on the free GitHub plan — 2-core uses those hours most efficiently.
    4. Set idle timeout. In Org Settings → Codespaces → Idle timeout, set to 30 minutes. A Codespace that sits idle will auto-suspend after 30 min, saving hours. Students can resume instantly by reopening the tab.
    5. Invite students to the org. Go to Org Settings → Members → Invite members. You'll need their GitHub usernames. Do this after they create accounts on Day 1 — or beforehand if you collect usernames via the Google Form. Students get an email and must accept.
    ⚠
    Codespaces hours budget check. Free GitHub accounts get 60 hours/month of 2-core Codespaces. With 13 sessions × 80 minutes = ~17.3 hours per student per month. You're well within budget. But if a student leaves a Codespace running overnight (common!), they'll drain it fast. The 30-min idle timeout protects against this.

    B · GitHub Classroom setup

    GitHub Classroom is what automates repo creation for each student. When a student clicks your assignment link, GitHub automatically creates a private repo for them, pre-loaded with your starter template.

    1. Go to classroom.github.com and sign in. Click New Classroom. Connect it to your org (the one you just set up). Name it "VibeCoding EMSI 2026".
    2. Set up your roster. GitHub Classroom can import students from your GitHub org, or you can skip the roster — students self-identify when they accept the assignment. For simplicity: skip the roster, just share the invitation link. Students' repos will be named [assignment-prefix]-[github-username].
    3. Create Assignment 1: Personal Website.
      • Assignment title: project1-personal-site
      • Repo prefix: project1- (student repos become project1-jsmith)
      • Visibility: Private (students can only see their own repo)
      • Template repo: your starter template (see Section C below)
      • Enable Feedback PRs: No (too complex for this course)
      • Copy the invitation link — you'll paste this into Google Classroom
    4. Create Assignment 2: Interactive Project. Same settings, prefix project2-, same template repo. Copy this second invitation link too.
    5. How GitHub Pages auto-deploys. When a student pushes to main, GitHub Pages automatically rebuilds their site. You need to enable Pages on the template repo first:
      • In the template repo → Settings → Pages
      • Source: Deploy from a branch
      • Branch: main, folder: / (root)
      • Save. This setting is inherited by all repos created from the template.
    ℹ
    How to check student commits from GitHub Classroom: Go to classroom.github.com → your classroom → the assignment → click any student's name. You see their repo with full commit history. This is your gradebook for the 20% Git workflow grade.

    C · Starter template repo

    This repo is the blank canvas every student starts from. It needs to exist before you create GitHub Classroom assignments.

    1. Create a new repo in your org called vibecoding-2026-template. Initialize with a README.
    2. Create the folder structure: 
      vibecoding-2026-template/
  .devcontainer/
    devcontainer.json      ← the file from this package
  index.html               ← starter HTML (see below)
  README.md                ← brief instructions for students
  style.css                ← blank CSS file
  script.js                ← blank JS file
    3. Paste this into index.html: 
      <!-- VibeCoding EMSI 2026 — Your starter file -->
<!-- Replace EVERYTHING between the tags with your own content -->
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>Your Name — VibeCoding 2026</title>
  <link rel="stylesheet" href="style.css">
  <style>
    body { font-family: sans-serif; margin: 40px; background: #0a0a0a; color: #eee; }
    h1 { color: #ff6b35; }
  </style>
</head>
<body>
  <h1>Your Name Here</h1>
  <p>Tell us something about you.</p>
</body>
</html>
    4. Mark the repo as a template. In repo Settings → check "Template repository". Now GitHub Classroom can use it.
    5. Add the devcontainer.json to the .devcontainer/ folder. This file (included in this package) tells Codespaces what extensions to install — Live Preview, Codeium, Prettier, etc. Without it, students get a bare VS Code.

    E · Google AI Studio (student AI tool)

    No setup needed on your end. Students sign in with their personal Google accounts (not school accounts) at aistudio.google.com/prompts/new_chat. It's free. No credit card. No verification.

    ℹ
    Why personal accounts, not school accounts? School Google accounts may have domain restrictions that block third-party apps. Personal accounts work reliably. AI Studio does not store student data in any FERPA-relevant way — it's a chat interface they use themselves. The code they generate lives in their GitHub repo, not in AI Studio.

    Settings to tell every student at the start of each class

    SettingValueWhy
    ModelGemini 3 FlashBest quality on the free tier for code generation
    ThinkingMinimalFaster responses; full thinking is slower and overkill for HTML/CSS/JS
    Grounding / SearchOFFGrounding is for research, not coding. Leaving it on wastes quota and adds irrelevant citations

    Plan B: If AI Studio is slow or down, send students to gemini.google.com. Same model, same Google account, simpler interface. Works identically.

    F · Critical pre-launch test: School WiFi

    🚨
    This is the highest-risk item in the entire course. GitHub Codespaces uses WebSockets to stream the VS Code interface. If the school network blocks WebSockets, students will see a loading spinner that never resolves. Do this test before Day 1 — not on Day 1 itself.
    1. Use a school-issued device (MacBook Air) on the school WiFi network. Do not use a personal device or a hotspot — the test must replicate exactly what students will experience.
    2. Go to github.com and open a repo. Any repo. Click the green Code button → Codespaces → Create codespace.
    3. Wait 90 seconds. You should see VS Code loading in the browser. If it loads: you're good. If it shows a spinning wheel that never resolves, or an error about connection: WebSockets are blocked.
    4. If blocked: Contact your IT department. Ask them to whitelist *.githubpreview.dev, *.github.dev, and WebSocket connections to GitHub. Alternatively, see the Contingency tab for an offline-first backup plan.

    D2
    Git loop + debugging exercise
    add → commit → push → Pages live for first time
    Tue Jun 9
    Phase 0
    ▶
    ℹ
    Goal: Every student runs git add . && git commit -m "first commit" && git push and sees their live GitHub Pages URL. This is the commit loop they'll use every day for 3 weeks.
    T+0
    Catch up stragglers (10 min)
    Any student who didn't finish Day 1 setup gets it done now with TA help. Everyone else: open their Codespace (if it went idle, click Resume — takes ~10 sec).
    T+10
    The terminal (5 min)
    Show how to open the terminal in VS Code: top menu → Terminal → New Terminal. A panel appears at the bottom. Explain: "This is how you talk directly to your computer using text commands." Show that the prompt shows the current folder. That's it for now.
    T+15
    First git commit (15 min)
    Everyone types these three commands in the terminal. Write them on the board. Walk through each one:

    git add . — "Stage everything. The dot means all files."
    git commit -m "first commit" — "Take a snapshot. Give it a message."
    git push — "Send the snapshot to GitHub."

    They should see output after push that says something like "Branch main set up to track origin/main." That means it worked.
    T+30
    GitHub Pages live (5 min)
    Have students go to their GitHub repo page (github.com/[org]/[their-repo-name]). Click Settings → Pages. They should see a green banner with their URL: https://[org].github.io/[repo-name]. Click it. Their site is live on the internet. First one who gets there: cheer.
    T+35
    Shared debugging exercise (35 min)
    This is the most important teaching moment of Phase 0. You provide everyone the same broken HTML file (see the debugging exercise file in this package). It has 5 intentional bugs — missing closing tags, wrong attribute syntax, a broken link, a CSS rule with a typo, and a JavaScript syntax error. Students paste it into their index.html and try to fix all 5 bugs using AI Studio. They can ask AI Studio to explain what's wrong. The goal is: learn to read an error, form a clear question, paste the code, get a fix, understand why it works. Circulate and ask students "what does that error mean?" before they ask AI. Force the thinking step first.
    T+70
    Commit the fixed file (10 min)
    Everyone runs the commit loop again on their fixed code. Message: "fixed all 5 bugs". Push. Check Pages updated. This is their second commit. Write the date and both commit messages somewhere they can reference — later they'll see what a good commit history looks like vs a bad one.
    ✓
    Checkpoint 0 complete: Every student has (1) a live GitHub Pages URL, (2) at least 2 commits, (3) has successfully run the git loop. Anyone missing these three things needs to complete them before Day 3.

You need to keep students on task and troubleshoot the most common problems. The students know more about coding than you need to — your job is logistics and support.

    What to do each day (simplified)

    Start of every class (first 5 minutes)

    • Tell students to open their Codespace. (github.com → their repo → Code → Codespaces → Resume)
    • Tell students to open AI Studio in a new tab. (aistudio.google.com)

    • Tell students to open Google Classroom for today's assignments

    End of every class (last 10 minutes)

    • If a student can't push: help them open terminal, type the commands one by one
    • Check that Google Classroom assignments for today are submitted

    Top 10 problems you'll see and how to fix them

    #What student saysWhat to do
    1"My Codespace won't load / is stuck"Hard-refresh the tab (Ctrl+Shift+R). If still stuck: go to github.com/codespaces, find their codespace, click the three dots → Stop codespace. Wait 10 sec. Then Resume. Takes 30-60 sec to restart.
    2"I can't see my changes in Live Preview"Make sure they saved the file (Ctrl+S). Check Live Preview is actually open — there's a small browser icon in the VS Code status bar at the bottom. Click it to open.
    3"Git push is asking for a username/password"This means authentication broke. Tell them to go to their GitHub repo in the browser → Code → Codespaces → delete this Codespace → create a new one. Takes 60 sec. Their code is saved in the repo — they won't lose work they already pushed.
    4"I pasted the AI code but nothing changed"Make sure they pasted it into the right file (index.html) and saved with Ctrl+S. Also: did they paste correctly? Sometimes they paste in the wrong place and create broken HTML. Have them select all (Ctrl+A), delete, then paste the full code fresh.
    5"AI Studio isn't giving me good code"Ask them to read their prompt out loud. It's probably too vague. Have them add: "Return only the changed code, not the full file." Or: "Explain what's wrong, then show the fix." Prompt quality = output quality.
    6"My site is broken on GitHub Pages but looks fine in Live Preview"This is usually a file path issue (a link to a file that doesn't exist). Tell them to push all their files and wait 60 seconds for Pages to rebuild. If still broken: the issue is a broken link or missing file — have them describe what's broken to AI Studio with the error.
    7"I accidentally deleted my code"Don't panic. If they had committed before: open terminal, type git log, find the last good commit message, type git checkout HEAD -- index.html. If they never committed: it may be gone. Ask them to re-open AI Studio and start from the last version they remember.
    8"I don't know what to do next"Ask: "What did you plan to build?" Open their idea ticket in Google Classroom. If they've finished their plan, suggest: add a color change, add more text, add a button. If totally stuck: have them type in AI Studio: "I'm building a [describe project]. What's the simplest next thing I could add?"
    9"AI Studio says 'rate limit' or 'too many requests'"Wait 60 seconds. Don't spam the refresh. They have 1,500 requests/day — they won't hit this limit from normal use. If they keep hitting it: they're probably submitting the same broken prompt over and over. Help them write one good prompt instead of many bad ones.
    10"The JavaScript doesn't work"Open the browser console: right-click anywhere on the Live Preview → Inspect → Console tab. Red text = errors. Have them copy the red error text and paste it into AI Studio with: "I got this error: [paste error]. Here is my code: [paste code]. What's wrong?"

    Contingency Plans

    Plan B: School WiFi blocks Codespaces (WebSockets blocked)

    🚨
    If students see a spinner that never resolves when opening a Codespace, WebSockets are blocked. This is the most catastrophic single failure mode. Here's how to survive it.

    Immediate fix for IT: Ask them to whitelist *.github.dev, *.githubpreview.dev, and allow WebSocket connections on port 443 to those domains. This often resolves the same day.

    If IT can't fix it same day — offline fallback:

    1. Download VS Code on each MacBook Air. It's free: code.visualstudio.com. Install it. This may be blocked by MDM — contact IT if so, or request a teacher-installed exception.
    2. Students work locally. They create a folder on their desktop. Open VS Code. Open that folder. Create index.html. The Live Preview extension still works locally — it's built into VS Code.
    3. Git still works. VS Code has a built-in Git panel (the branch icon in the left sidebar). Students can still add/commit/push to GitHub. GitHub Pages still deploys.
    4. The only thing lost: Cloud sync. Students' code now lives on their MacBook, not in the cloud. If the MacBook breaks, they lose uncommitted work. Emphasize: push every 20 minutes.

    Plan C: GitHub is down

    Check status.github.com. GitHub has ~99.9% uptime. If it's down: run a non-coding session.

    • Students plan/sketch their project on paper (idea ticket + wireframe)
    • Teach a 30-minute prompt engineering workshop using AI Studio (which doesn't require GitHub)
    • Code review session: students share screens and explain their code to a neighbor

    Plan D: AI Studio is down

    ⚠
    First: try gemini.google.com. Same model, different URL, usually works when AI Studio has a front-end issue.

    If both are down:

    • Students work on manual HTML/CSS edits — changing colors, text, adding/removing sections
    • Show them basic HTML tags directly: they can add a new paragraph, change a font size, add a link
    • This is actually a good backup lesson: "what can you do without AI?"
